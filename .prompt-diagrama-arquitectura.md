## Contexto del Proyecto

Es una aplicación web para que docentes de la ESPOCH (Escuela Superior Politécnica de Chimborazo) gestionen notas de estudiantes. Se integra con OASIS (el sistema académico institucional) vía SOAP para importar nóminas, horarios, materias y notas.

## Base de Datos (PostgreSQL)

5 tablas con columnas JSONB para datos flexibles:

- `docente(id, email PK, nombre, cedula, rol, password_hash)`
- `asignacion(id PK, docente_email FK, carrera, asignatura, pao, paralelo, data JSONB)`
- `configuracion(id PK, owner_email FK, carrera, asignatura, pao, data JSONB, saved_at)`
- `config_estudiantes(config_id PK FK, data JSONB)`
- `config_notas(config_id PK FK, data JSONB)`

La DB es **opcional**: si no hay PostgreSQL configurado, la app funciona con localStorage.

## Decisiones Arquitectónicas Clave

1. **BFF Pattern**: El navegador nunca habla SOAP ni toca la base de datos directamente.
2. **React decorativo**: React solo renderiza el HTML inicial; toda la interactividad es Vanilla JS con innerHTML y document.getElementById().
3. **Doble backend**: Dos BFFs independientes (PHP activo, Node legacy) con la misma API REST.
4. **Mock data siempre lista**: Cada operación SOAP tiene try-catch con fallback a datos mock.
5. **SOAP sin librerías**: Ambos BFFs implementan SOAP manualmente (XML templates + HTTP puro).
6. **Resolución fuzzy de nombres**: Las funciones usan coincidencia sin acentos/tildes para mapear nombres humanos a códigos OASIS.
7. **Store sincrónico**: Protocolo GET/PUT /api/store para sincronizar datos entre localStorage y PostgreSQL.

### 1. Diagrama Mermaid.js
Genera un diagrama de arquitectura completo que incluya:

- BFF (PHP activo y Node legacy)

- Almacenamiento local (localStorage)

### 2. Análisis crítico y recomendaciones
Evalúa:

**Debilidades detectadas:**
- React se usa solo como "template engine", toda la lógica está en JS vanilla con manipulación directa del DOM → difícil de mantener y testear
- Dos BFFs con la misma API → duplicación de código
- Implementación SOAP manual → frágil, sin tipado, sin WSDL validation
- Sin tests automatizados (ni unitarios, ni integración, ni e2e)
- Sin TypeScript → propenso a errores en caliente
- Sin CI/CD pipeline
- El BFF Node.js no tiene manejador de errores para el caso `EADDRINUSE` ni reconexión
- Variables de entorno en `.env` sin validación al arranque
- El frontend hace una sola llamada `getEstudianteFull` pero el backend aún hace múltiples SOAP secuenciales para encontrar la carrera del estudiante
- Web Worker de anomalías no tiene comunicación bidireccional estructurada con el main thread

**Recomendaciones esperadas:**
- Migrar a un framework frontend real (Next.js, o al menos React puro con hooks)
- Unificar los BFFs (elegir uno, eliminar el otro)
- Agregar tests (Vitest para frontend, PHPUnit para backend)
- Tipar con TypeScript
- CI/CD con GitHub Actions
- Mejorar el manejo de errores y timeouts en SOAP
- Agregar validación de esquema en el BFF
- Centralizar configuración (eliminar duplicación entre `.env` files)
